Layout

Comrade generally uses a 12 column grid with media queries for responsive layout.

Most recently we've been using Twitter Bootstrap as a starting point for projects.

Tips for working with a framework (regardless of starting framework)

  • Familiarize yourself with framework components/features before beginning - most standard site features will be included with the framework.
  • Rely on framework for basic structure (columns/grid)
  • Bootstrap is highly customizable and has many built in features (as do most frameworks.) Don't waste time coding your own version of a feature if the frameowrk has done the work and heavily tested it already! In many cases the component can be customized to meet any needs.
  • Keep framework CSS and javascript separate in your development environment in case an upgrade is needed - the files can be minified and concatenated before deployment.

Mobile Optimization

<meta name="viewport" content="width=device-width">

Media Queries (try to limit number of breakpoints)

  • With so many devices it is impossible and inadvisable to try and target any specifically. Breakpoints should fit the design not vice versa.
  • Limiting yourself to 2 or 3 breakpoints (at most) allows the designer and developer to focus more on the whole site instead of focusing on specific device views.
  • Targeting specific devices can lead to weird layouts in unexpected places.

Images lazy loading, srcset, compression

  • With the advent of srcset lazy-loading is less necessary, however depending on the content and length of the page it can still be a great asset on mobile or web.
  • Combining lazy loading with a CDN and srcset is a good way to minimize page load times on image intensive sites.
  • We recommend unveil.js when you just need to lazy load.
  • There are lots of image compression tools online - here are a couple: compressor, kraken

Minification and concatenation

  • Minification of CSS, Javascript and HTML helps with page load time, which is especially important to consider on mobile where users may be connecting over spotty shared wifi, or using slow cell connections.
  • Concatenation of files also helps to limit server calls which is important for the same reason.

Coding Style

Indent using a tab representing 4 spaces for HTML, CSS and Javascript

When using an existing code base use the spoacing set by the previous coder. Consistency is best!

HTML

<!—  Add breaks for readability when necessary —>
<p>
    Lorem ipsum dolor sit amet, consectetur adipiscing elit. Cras fringilla
    ibero id quam semper semper. Nulla a pulvinar justo. Mauris a velit sed
    ante laoreet vehicula. Etiam ac semper mi. Etiam neque lacus, hendrerit
    a sollicitudin sagittis.
</p>
<ul>
    <li>This is ok because even though it's pretty long, it's really too short to be on 2 lines.</li>
</ul>
  • Keep markup at a reasonable width and add breaks for readability when necessary.
  • With short lines of content, the tag and content can stay on the same line.
  • Using your favorite text editor or IDE find a plugin that will keep your formatting consistent. Many editors have plugins or settings that will prettify your code on save or with a keyboard shortcut. If this isn't an option check out prettifier.
  • Always follow best practices for semantic and accessible markup. (See our accessibility guide for examples.)

CSS

  • Comrade often uses LESS or SASS which is transpiled into CSS. This is determined by project and client needs and we are well versed in each.
  • Use OOCSS/SMACSS as your guiding CSS principles
    • SMACSS Site
    • Read about OOCSS on smashingmagazine.com
    • Read about OOCSS on CSSWizardry
  • Use class selectors as opposed to IDs
  • Don't use !important unless you have a really good reason. If you feel like you need to use it, think very carefully about why that would be the case, then refactor your CSS to solve the problem. (But seriously, there are sometimes reasons to use it, just make sure you have a good reason.)
  • Don't use <br>, <strong>, <em>, etc. just for styling. (Notice that the previous statement is wrapped in a strong tag - that's because this is important.)
  • We use auto-prefixer to take care of vendor specificity when possible. If we are working on a project where we would not have access to something like auto-prefixer we will add vendor prefixes as necessary.
  • Use rems when possible when dealing with text (font sizes, line-height, margins, padding, etc.)
  • Use percents for box level elements
  • Don’t mix and match measurement units unless you have a good reason
    (for example: don't use px AND pts at the same time)
  • Don't use pts unless you are working on a PRINT style sheet. The web is rendered in pixels.
  • When joining an in progress project conform to the establised coding standards, i.e. indentations, units, etc.
  • Formatting:
    • Each selector lives on its own line followed by the opening brace
    • Each declaration lives on its own indented line
    • Group the declarations in a logical way, whether this means alphabetically or by declaration type is not important as long as you are consistent.
    • Each property has a space between the colon and value ends with a semi-colon
    • The closing brace should be on its own line at the same indentation as the selector
h1 {
    font-size: 1.125em;
    position: relative;
    margin-bottom: 1em;
    padding-bottom: .375em;
}

Javascript

  • Formatting:
    • Use tabs for spacing. A tab is expected to represent the width of 4 spaces.
    • When joining an in progress project conform to the establised coding standards, i.e. indentations, units, etc. (i.e. if previously 2 spaces were used stick with this)
    • Place the brace on the same line as control statements and same line as function declarations.
    • If more than one action is called on a jQuery object indent those actions.
var myFunction = function () {
    var newText = 'Hi There!',
        newDiv = $('#greeting');

    $('.selector').text(newText)
   		  .addClass('lineup')
		  .appendTo(newDiv);
};
  • Keep javascript code in closures. This minimizes side effects and unintended interactions between different JavaScript plugins and snippets.
  • Variable names, functions, classes and plugin names should all be in the lowerCamelCase format.
  • Variable names must be meaningful (except when they aren’t).
  • Include lots of comments please.
  • If a jQuery element is going to be used more than once, set it to a JavaScript variable to prevent calling the DOM every time it needs to be used (this is known as object caching and it just means we don't need to search for that element over and over again)
  • Always use JSHint (JSLint is too bossy)
  • console.log is built into Chrome, Safari, Opera, and Firebug. However, make sure there are no calls to console.log in any file when launched, as they will break in older versions of IE.

Browser Support

We use StatCounter to evaluate browser usage and support. Our standard list of supported browsers is updated regularly to reflect changing usage.

  • Internet Explorer, older browsers and graceful degradation
    • For older less Standards Compliant browsers we support the concept of graceful degradation. This means that even though the site may not look exactly the same in IE8 as it does in the latest version of chrome we will make sure that it looks great in both.
    • Graceful degradation may mean that a background may be a solid color in IE8, but a gradient in Safari. It could mean the opacity is solid for an element in an older browser and translucent in a modern browser. Aesthetics are not sacrificed in the older browsers, they may just miss a few flourishes.
    • In many cases letting the browser determine the level of enhancement provides a smoother experience for someone using an older and slower computer/browser.

Typography

  • Font Stacks - All fonts MUST have fallbacks to common and default fonts. No exceptions!
h1 {
    font-family: roboto, arial, sans-serif;
}

Web Fonts

We use typekit, google fonts or fonts.com for web font support.

Icons

At comrade we often use icomoon.io for providing custom icon fonts in our projects. When this is not the case we will use SVGs or a spritesheet.

  • All icons should be contained in a single sprite sheet.
  • If using SVG icons there must be a fallback for supported legacy browsers.

Validation

  • All HTML, CSS and Javascript should be valid.
  • Markup should be well-formed and contain all required attributes.
  • HTML should validate with the W3C Markup Validation Service.
  • Stylesheets should validate with the W3C CSS Validation Service.
  • Javascript should validate with JSHint.
