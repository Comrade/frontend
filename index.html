Coding Style

Indent using a tab representing 4 spaces for HTML, CSS and Javascript

HTML

<!—  Add breaks for readability when necessary —>
<p>
    Lorem ipsum dolor sit amet, consectetur adipiscing elit. Cras fringilla
    ibero id quam semper semper. Nulla a pulvinar justo. Mauris a velit sed
    ante laoreet vehicula. Etiam ac semper mi. Etiam neque lacus, hendrerit
    a sollicitudin sagittis.
</p>
<ul>
    <li>This is ok because even though it's pretty long, it's really too short to be on 2 lines.</li>
</ul>
  • Keep markup at a reasonable width and add breaks for readability when necessary.
  • With short lines of content, the tag and content can stay on the same line.
  • Always follow best practices for semantic and accessible markup. (see accessibility section for code examples)

CSS

  • Use OOCSS/SMACSS as your guiding CSS principles
    • SMACSS Site
    • Read about OOCSS on smashingmagazine.com
    • Read about OOCSS on CSSWizardry
  • Use class selectors as opposed to IDs
  • Don't use !important unless you have a really good reason. If you feel like you need to use it, think very carefully about why that would be the case, then refactor your CSS to solve the problem.
  • We use auto-prefixer to take care of vendor specificity when possible. If we are working on a site where we would not have access to something like auto-prefixer we will account for vendor prefixes as necessary.
  • Use ems for text
  • Use percents for box level elements
  • Don’t mix and match measurement units unless you have a good reason
    (for example: don't use px AND pts at the same time)
  • Formatting:
    • Each selector lives on its own line followed by the opening brace
    • Each declaration lives on its own indented line
    • Group the declarations in a logical way, whether this means alphabetically or by declaration type is not important as long as you are consistent.
    • Each property has a space between the colon and value ends with a semi-colon
    • The closing brace should be on its own line at the same indentation as the selector
h1 {
    font-size: 1.125em;
    position: relative;
    margin-bottom: 1em;
    padding-bottom: .375em;
}

Javascript

  • Formatting:
    • Use tabs for spacing. A tab is expected to represent the width of 4 spaces.
    • Place the brace on the same line as control statements and same line as function declarations.
    • If more than one action is called upon a jQuery object indent those actions.
var myFunction = function () {
    var newText = 'Hi There!',
        newDiv = $('#greeting');
		
    $('.selector').text(newText)
   		  .addClass('lineup')
		  .appendTo(newDiv);
}
  • Keep javascript code in closures. This minimizes side effects and unintended interactions between different JavaScript plugins and snippets.
  • Variable names, functions, classes and plugin names should all be in the lowerCamelCase format.
  • Variable names must be meaningful (except when they aren’t).
  • Include lots of comments please.
  • If a jQuery element is going to be used more than once, set it to a JavaScript variable to prevent calling the DOM every time it needs to be used (this is known as object caching)
  • For graceful degradation, always make sure that the DOM and initial CSS reflect the state you’d like the site in when JavaScript is not available. Even if content will initially be hidden, consider how the site should look without JavaScript available and hardcode your HTML and CSS to match that.
  • Always use JSHint (JSLint is too bossy for us)
  • console.log is built into Chrome, Safari, Opera, and Firebug. However, make sure there are no calls to console.log in any file when launched, as they will break in IE.

Typography

  • Font Stacks - All fonts MUST have fallbacks to common and default fonts. No exceptions!
h1 {
    font-family: roboto, arial, sans-serif;
}

Web Fonts

We use typekit, google fonts or fonts.com for web font support.

Icons

  • All icons should be contained in a single sprite sheet.
  • If using SVG icons there must be a fallback for non-supporting browsers.

Validation

  • All HTML, CSS and Javascript should be valid.
  • Markup should be well-formed and contain all required attributes.
  • HTML should validate with the W3C Markup Validation Service.
  • Stylesheets should validate with the W3C CSS Validation Service.
  • Javascript should validate with JSHint.
